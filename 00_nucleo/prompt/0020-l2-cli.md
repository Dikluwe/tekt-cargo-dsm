# Prompt: L2 CLI da lente (catálogo + binário) sob ADR-0002

**Camada**: L2 — Casca
**Criado em**: 2026-06-01 (reescrito após formalização do ADR-0002 no Tekt)
**Estado**: `PROPOSTO`
**Decisões de origem**:
- **Tekt ADR-0002** (apresentação como responsabilidade exclusiva de L2): toda
  apresentação (texto, formato) mora em L₂; código nunca contém literais de
  apresentação; conteúdo do catálogo é dado de L₂.
- Decisões do autor sobre a CLI: modo focado apenas, entrada por `--pacote`
  ou `--grafo`, alvo por path ou id, output JSON default com `--text` e
  `--verbose`, clap como parser, códigos de saída simples (0/1/2), tradução
  id→path mostrada na saída, catálogo em crate separado, mensagens como
  templates com placeholders.
- Decisão sobre o `Display` do `ErroLente` em L4 (laudo 0019): permanece como
  representação técnica do tipo; **a CLI não o usa para apresentação** — usa
  o catálogo.
**Terceiro e último de três prompts da composição** (L3 invocador → L4 wiring
→ **L2 CLI**).
**Pré-requisito**: L3 consolidado (laudo 0018), L4 wiring (laudo 0019). Crate
`lente_wiring` expõe `calcular_raio_de_alvo`, `FonteGrafo`, `AlvoBusca`,
`ErroLente`.
**Arquivos afetados**: dois crates novos — `02_shell/catalogo/` (lib) e
`02_shell/cli/` (bin).

---

## Contexto

O L4 (`lente_wiring`) expõe a função de composição pura. O L2 CLI é a primeira
boca da lente: traduz argumentos da linha de comando em chamada à L4, formata
a saída e a escreve em stdout/stderr.

Este prompt nasce sob o **ADR-0002 do Tekt**, formalizado depois do laudo
0019. Isso significa que a CLI **não pode** ter strings de apresentação
inline em handlers ou no main. Toda mensagem ao usuário (saída, erro, ajuda
adicional) vem de um catálogo externalizado da lógica de orquestração.

---

## Restrições estruturais

- **L2 — admite I/O de canal externo** (terminal: args, stdout, stderr,
  códigos de saída).
- **Importa L4 e L1**: `lente_wiring` para chamar a composição, `lente_core`
  para tipos que aparecem (`Path`, `Raio`, `Classificacao`).
- **Não importa L3 nem outros L1** (`lente_investiga`, `lente_resolve`) —
  tudo desce via L4.
- **Aderência absoluta ao ADR-0002**: nenhum literal de apresentação fora do
  catálogo. O linter desta regra é manual (revisão), mas o desenho impede o
  problema por construção — o crate `lente_cli` só constrói strings via
  chamadas ao catálogo.
- **`lente_core` continua puro**: este prompt não toca nada do L1.

---

## Parte 1 — Crate `lente_catalogo` em `02_shell/catalogo/`

### Propósito

Centraliza todas as mensagens que a lente emite. **Conteúdo é dado de L₂**
(ADR-0002, regra 3). Nesta primeira versão, conteúdo vive como constantes
e templates dentro do código Rust do crate — quando/se o autor decidir
externalizar (arquivo `.toml`, `.ftl`), o **carregamento** vira trabalho de
L₃, mas o conteúdo continua sendo L₂.

### Estrutura

```
02_shell/catalogo/Cargo.toml  — lib, sem deps externas
02_shell/catalogo/src/lib.rs  — API pública do catálogo
```

Sem deps externas — só stdlib. Crate independente, importável por qualquer
boca L2 futura.

### API pública: templates com placeholders

Decisão do autor: mensagens como **templates com placeholders**. Forma
concreta sugerida (gerador pode ajustar nomenclatura):

```rust
/// Templates do catálogo. Cada constante é um template com placeholders
/// `{nome}` substituídos via método `render`.
pub struct Template(&'static str);

impl Template {
    /// Substitui placeholders pelos valores fornecidos.
    /// Placeholders ausentes do mapa permanecem como `{nome}` literal
    /// (degradação visível, não panic).
    pub fn render(&self, vars: &[(&str, &str)]) -> String { ... }
}

// === ERROS ===
pub const ERRO_FORK_AUSENTE: Template = Template(
    "Não foi possível invocar o cargo-modules: {detalhe}"
);
pub const ERRO_JSON_INVALIDO: Template = Template(
    "JSON do fork não pôde ser desserializado: {detalhe}"
);
pub const ERRO_ALVO_INEXISTENTE: Template = Template(
    "Alvo '{alvo}' não existe no grafo"
);
pub const ERRO_ID_INEXISTENTE: Template = Template(
    "Nó com id {id} não existe no grafo"
);
pub const ERRO_RESOLUCAO: Template = Template(
    "Falha na resolução de colisão em '{path}': {detalhe}"
);
pub const ERRO_GENERICO_PIPELINE: Template = Template(
    "Falha no pipeline: {detalhe}"
);

// === SAÍDA (campos do JSON e rótulos do texto) ===
pub const ROTULO_ALVO: &str = "Alvo";
pub const ROTULO_ALVO_PEDIDO: &str = "Alvo pedido";
pub const ROTULO_ALVO_RESOLVIDO: &str = "Alvo resolvido";
pub const ROTULO_CLASSIFICACAO: &str = "Classificação";
pub const ROTULO_DIRETOS: &str = "Impacto direto";
pub const ROTULO_TRANSITIVOS: &str = "Transitivo";
pub const ROTULO_IMPACTADOS: &str = "Impactados";

// === CLASSIFICAÇÕES (strings legíveis para Classificacao do lente_core) ===
pub const CLASSIFICACAO_ISOLADO: &str = "Isolado";
pub const CLASSIFICACAO_FOLHA: &str = "Folha";
pub const CLASSIFICACAO_BASE: &str = "Base";
pub const CLASSIFICACAO_INTERMEDIARIO: &str = "Intermediário";

// === HELP / ABOUT (clap) ===
pub const ABOUT_CLI: &str = "O que quebra se eu mexer aqui?";
pub const HELP_GRAFO: &str = "JSON pronto (gerado pelo fork cargo-modules 0.27.0)";
pub const HELP_PACOTE: &str = "Nome do pacote — a lente invoca o fork internamente";
pub const HELP_ALVO: &str = "Path do alvo no grafo (ex.: 'ErroRaio::<Display>::fmt')";
pub const HELP_ALVO_ID: &str = "Id do alvo no grafo (alternativa ao --alvo)";
pub const HELP_TEXT: &str = "Saída em texto humano-legível (default é JSON)";
pub const HELP_VERBOSE: &str = "Inclui lista completa de itens impactados";
```

### Sobre `render` (decisão do gerador, com diretriz)

Implementação simples de substituição de placeholders. Não usar regex ou
crate de templates — stdlib basta. Sugestão: percorrer a string, achar
`{nome}`, substituir. Placeholders ausentes do mapa permanecem como
`{nome}` literal (degradação visível). Não panic.

Se o gerador quiser uma forma mais Rust-idiomática (ex.: tipo de "args"
estruturado em vez de slice de tuplas), ok — mas mantenha simples.

### Testes do catálogo

- `render` substitui um placeholder.
- `render` substitui múltiplos.
- Placeholder ausente do mapa permanece literal.
- Constantes têm valores não-vazios (sanidade).

## Parte 2 — Crate `lente_cli` em `02_shell/cli/` (binário)

### Estrutura

```
02_shell/cli/Cargo.toml         — binário "lente", deps:
                                   lente_wiring, lente_core, lente_catalogo
                                   (por path); clap (derive); serde_json
02_shell/cli/src/main.rs        — ponto de entrada
02_shell/cli/src/args.rs        — definição clap (usa lente_catalogo para help)
02_shell/cli/src/saida.rs       — formatação (usa lente_catalogo para tudo)
02_shell/cli/src/erro.rs        — tradução ErroLente → mensagem do catálogo
```

### Argumentos (clap derive, usando catálogo para todas as strings)

```rust
#[derive(Parser)]
#[command(name = "lente", about = lente_catalogo::ABOUT_CLI)]
struct Args {
    #[arg(long, conflicts_with = "pacote", help = lente_catalogo::HELP_GRAFO)]
    grafo: Option<PathBuf>,

    #[arg(long, conflicts_with = "grafo", help = lente_catalogo::HELP_PACOTE)]
    pacote: Option<String>,

    #[arg(long, conflicts_with = "alvo_id", help = lente_catalogo::HELP_ALVO)]
    alvo: Option<String>,

    #[arg(long = "alvo-id", conflicts_with = "alvo", help = lente_catalogo::HELP_ALVO_ID)]
    alvo_id: Option<usize>,

    #[arg(long, help = lente_catalogo::HELP_TEXT)]
    text: bool,

    #[arg(long, help = lente_catalogo::HELP_VERBOSE)]
    verbose: bool,
}
```

Validar manualmente após o parse (clap só cobre `conflicts_with`):
- Exatamente um de `--grafo`/`--pacote` (senão erro 2 via mensagem própria).
- Exatamente um de `--alvo`/`--alvo-id` (senão erro 2).

### Fluxo do `main()`

1. Parse args via clap.
2. Validar exclusividades (`--grafo`/`--pacote`, `--alvo`/`--alvo-id`).
3. Construir `FonteGrafo` (lendo arquivo se `--grafo`; passando nome se
   `--pacote`).
4. Construir `AlvoBusca` (path ou id).
5. Chamar `lente_wiring::calcular_raio_de_alvo(fonte, alvo)`.
6. Tratar resultado:
   - `Ok(raio)`: formatar via `saida` conforme `--text` e `--verbose`,
     escrever em stdout. **Tradução id→path**: se o usuário passou `--alvo-id`
     e o `Raio.alvo` (path no grafo resolvido) é diferente do que ele pediu
     conceitualmente, incluir ambos na saída (campo `alvo_pedido` e
     `alvo_resolvido` no JSON; duas linhas no texto). Sair com 0.
   - `Err(erro_lente)`: traduzir via `erro` (módulo que casa as variantes
     do `ErroLente` e seleciona o template do catálogo), imprimir em
     stderr, sair com 1.

### Tradução id→path (importante — decisão do autor)

Quando o usuário passa `--alvo-id N`:
- O L4 resolve o id no grafo já resolvido e devolve `Raio` com o path
  **final** (após resolução de colisões).
- A CLI guarda também o id pedido. Se o path final do `Raio` é diferente
  do que estaria "obviamente vinculado" ao id (sempre é, em colisões
  resolvidas — o nome mudou), a saída inclui:
  - JSON: `{"alvo_pedido": "id=20", "alvo_resolvido":
    "ErroRaio::<Display>::fmt", ...}`.
  - Texto: duas linhas (`Alvo pedido: id=20`, `Alvo resolvido:
    ErroRaio::<Display>::fmt`).

Quando o usuário passa `--alvo PATH`, e o path informado existe no grafo
**final** sem ter sido renomeado, mostra só "Alvo: <path>" (sem distinção).

### Saída em quatro modos (matriz `--text` × `--verbose`)

**JSON resumo (default)**:
```json
{"alvo": "...", "classificacao": "Folha", "diretos": 0, "transitivos": 0}
```
(Com `alvo_pedido`/`alvo_resolvido` se for `--alvo-id`.)

**JSON verboso (`--verbose`)**: adiciona `"impactados": [...]` com lista de
paths (ordem: decisão do gerador, registrar — sugestão alfabética).

**Texto resumo (`--text`)**:
```
Alvo:           ErroRaio::<Display>::fmt
Classificação:  Folha
Impacto direto: 0 itens
Transitivo:     0 itens
```

**Texto verboso (`--text --verbose`)**: o resumo + seção "Impactados:" com
um item por linha indentado.

Todos os rótulos vêm do catálogo (`ROTULO_ALVO`, `ROTULO_CLASSIFICACAO`,
etc.). Nada inline.

### Tradução de erro (módulo `erro.rs`)

Função pública que recebe `&ErroLente` e devolve `String` (mensagem
traduzida):

```rust
pub fn traduzir(erro: &lente_wiring::ErroLente) -> String {
    use lente_catalogo as cat;
    match erro {
        ErroLente::Fork(ef) => cat::ERRO_FORK_AUSENTE.render(&[
            ("detalhe", &format!("{}", ef))  // Display técnico do L4, ok como detalhe interno
        ]),
        ErroLente::Adaptador(ea) => cat::ERRO_JSON_INVALIDO.render(&[
            ("detalhe", &format!("{}", ea))
        ]),
        ErroLente::Raio(_) => cat::ERRO_ALVO_INEXISTENTE.render(&[
            ("alvo", "<alvo informado>")  // ver nota
        ]),
        ErroLente::IdInexistente(id) => cat::ERRO_ID_INEXISTENTE.render(&[
            ("id", &id.to_string())
        ]),
        ErroLente::Resolucao(er) => cat::ERRO_RESOLUCAO.render(&[
            ("path", "<path>"),
            ("detalhe", &format!("{}", er))
        ]),
    }
}
```

**Nota sobre `Display` do erro interno**: o `Display` técnico do `ErroLente`
e dos erros internos (`ErroFork`, `ErroAdaptador`, etc.) é cru, mas serve
como "detalhe" embutido no template do catálogo. O catálogo dá a moldura
("Não foi possível invocar..."), o `Display` técnico dá o detalhe específico
(`I/O error: No such file`). É composição de duas camadas: o catálogo é
apresentação; o Display é informação técnica. Aceito.

Se uma variante de erro tem informação contextual útil (ex.: o alvo
informado pelo usuário, no caso de `Raio::AlvoInexistente`), passar essa
informação para a tradução exige conhecer o contexto — o `main()` pode
construir uma `ContextoErro` que carrega o alvo informado, e a função
`traduzir` recebe `(&ErroLente, &ContextoErro)`. Decisão do gerador.

```

Dado --grafo arquivo.json --alvo-id 20 --text --verbose
Quando executado
Então imprime texto com:

Quando executado

Quando executado a partir da raiz do projeto, com fork 0.27.0 instalado
Então invoca fork, roda pipeline, imprime JSON. Código 0.
(Teste #[ignore] se necessário para CI sem fork.)

Quando grep --include='*.rs' -E '"[A-Z][a-z]+|"erro|"falha' em 02_shell/cli/
```

- Os quatro modos de saída (JSON/texto × resumo/verboso).
- Tradução id→path mostrada quando aplicável.
- Cada variante de `ErroLente` produz mensagem do catálogo certa.
- Validação manual de exclusividades.
- E2E real (`#[ignore]` se necessário).

## O que NÃO entra (cascata futura)

- **Modo ranking**: `lente --pacote X --ranking`. Próximo prompt em sequência.
- **Modo agregado**: `lente --pacote X` (sem alvo). Em sequência depois.
- **Boca biblioteca**: expor `calcular_raio_de_alvo` para outros crates Rust
  usarem. Prompt próprio.
- **Boca servidor (LSP/HTTP)**: prompt próprio bem depois.
- **Carregamento de catálogo externo (arquivo `.toml`/`.ftl`, escolha de
  idioma)**: refinamento futuro. **O conteúdo continua sendo L₂**; só o
  carregamento vira trabalho de L₃ (ADR-0002 regra 3).
- **Cores no terminal**, paginação: refinamento de UX, não agora.

## Observação metodológica

Este prompt é o **primeiro componente do projeto-lente que nasce sob o
ADR-0002 do Tekt**. Aderência rigorosa é importante porque (i) o ADR é
novo e este componente é o primeiro teste prático dele; (ii) qualquer
violação inicial cria precedente para futuros componentes L2.

Se a aderência ao ADR for difícil em algum ponto específico (ex.: clap
tem mensagens embutidas em inglês que não controlamos), **registrar a
exceção no laudo** em vez de violar silenciosamente. Exceções
documentadas são aceitáveis; aderência implícita não.

Este é também o **primeiro momento em que a lente vai produzir saída
observável**. Esperar descobertas — formatação que parecia boa mas é
confusa em prática, valores que parecem suspeitos. Registrar tudo.
