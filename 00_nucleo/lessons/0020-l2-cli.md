## A lente rodou do terminal pela primeira vez

Saídas reais capturadas com o binário `lente` invocando o fork e atravessando
a cascata inteira ponta-a-ponta:

### `lente --pacote lente_core --alvo-id 36`

```json
{"alvo_pedido":"id=36","alvo_resolvido":"lente_core::domain::raio::ErroRaio::<Display>::fmt","classificacao":"Folha","diretos":0,"transitivos":0}
```

**Tradução id→path em produção**: o usuário pediu `id=36`; o pipeline
desserializou, detectou a colisão `ErroRaio::fmt` (Display+Debug), resolveu
para `<Display>`/`<Debug>` lendo `trait_` do nó (cascata do descritor inteira,
laudos 0012-0015 e 0019), e a CLI mostrou os dois nomes (id pedido + path
resolvido). É a primeira observação direta do mecanismo funcionando do terminal.

### `lente --pacote lente_core --alvo-id 36 --text --verbose`

```
Alvo pedido:	id=36
Alvo resolvido:	lente_core::domain::raio::ErroRaio::<Display>::fmt
Classificação:	Folha
Impacto direto:	0 itens
Transitivo:	0 itens

Impactados:
```

Rótulos do catálogo, formatação tabular. **Descoberta interessante**: o
método `fmt` do `Display` é classificado como **Folha** com zero impacto —
correto, porque nenhum item do `lente_core` *usa* `Display::fmt`
diretamente; ele é chamado pelo runtime Rust (formatter macros), que está
no sysroot e não conta como dependência "interna" no grafo do crate. É o
limite do que `cargo-modules` extrai: vê acoplamentos estruturais (assinaturas,
campos), não chamadas pelo formatter macro.

### `lente --pacote lente_core --alvo "nao::existe"` → stderr, exit 1

```
Alvo 'nao::existe' não existe no grafo
```

Mensagem em português, do catálogo. Código de saída 1 (erro de pipeline).

Todos os matches encontrados estavam **dentro de `#[cfg(test)] mod tests`**

produção). Fora dos testes: **zero literais de apresentação**. ADR-0002

### Exceção registrada

Mensagens do **clap** (em inglês — `error: the argument '--grafo' cannot
be used with '--pacote'`, `Usage: lente [OPTIONS]`, etc.) não são cobertas
pelo nosso catálogo. O clap tem seu próprio catálogo interno em inglês,
fora do nosso controle direto. Registrado como exceção aceitável; se um
dia for crítico traduzir, há crates como `clap_lex` ou monkey-patch do
contexto de tradução do clap.

## Decisões tácitas

### D1 — Dois crates: catálogo separado do binário

`lente_catalogo` é lib **sem deps externas**, importável por qualquer boca
L2 futura (LSP, HTTP, GUI). O `lente_cli` é o binário concreto. Separação
prevista pelo prompt; benefício real: o catálogo é o "contrato textual" do
projeto, reutilizável.

### D2 — `Template::render` linha-a-linha, stdlib only, UTF-8 safe

Parser simples (varre bytes, casa `{nome}`, substitui). Cuidado com UTF-8
multibyte (testado em `render_lida_com_utf8_multibyte`). Placeholder
ausente do mapa permanece literal `{nome}` — degradação visível, não panic
nem erro silencioso.

### D3 — `tem_traducao` mostra alvo_pedido+resolvido **só quando veio por id**

Quando o usuário passa `--alvo PATH`, o que ele pediu é o que ele pediu —
saída tem só o campo `alvo` (JSON) ou linha "Alvo:" (texto). Quando passa
`--alvo-id N`, mostra **sempre** os dois campos `alvo_pedido` + `alvo_resolvido`
(o N é opaco; o usuário precisa ver o path correspondente para entender o
resultado).

Decisão diferente do que o prompt sugeria ("se houve renomeação"): adotei
"sempre que veio por id, mostra os dois" porque (a) o usuário pediu por id,
provavelmente quer ver o path, e (b) detectar "renomeação" precisaria do
path original (que não temos no input — só temos o id).

### D4 — Validação manual de exclusividade de args

Clap cobre `--grafo` + `--pacote` (ambos presentes) via `conflicts_with`,
mas **não** cobre "nenhum dos dois informado". Validação manual em
`construir_fonte` e `construir_alvo` devolve `SaidaErro { codigo: 2 }`
com mensagem do catálogo (`ERRO_FONTE_NAO_INFORMADA`,
`ERRO_ALVO_NAO_INFORMADO`).

### D5 — `ContextoErro` simples: só `alvo_informado`

A tradução de `ErroLente::Raio(AlvoInexistente(Path))` substitui o `Path`
interno (que pode estar renomeado pelo pipeline, ou estar no formato
interno) pelo texto que o usuário **entende** — o `alvo_informado` da CLI
(path direto ou `"id=N"`). Mais útil para o usuário que o path interno.

### D6 — `main()` fino + `run()` testável

Padrão limpo para evitar `assert_cmd` como dep: o `main()` faz só
`Cli::parse()` + `run(cli)` + tradução do resultado em ExitCode. O `run`
é função pura testável (recebe `Cli`, devolve `Result<String, SaidaErro>`).
Os 7 testes do `main.rs` exercitam `run` diretamente — incluindo o E2E
`#[ignore]` contra o `lente_core` real.

### D7 — Diretos vs. Transitivos

Métricas mostradas na saída:
- **Impacto direto** = `raio.uses_entrada` (vizinhos imediatos por `Uses`).
- **Transitivo** = `raio.montante.len()` (todos os nós alcançáveis pelo BFS
  reverso).

Diferença didática: "quantos usam diretamente" vs. "quantos podem sentir,
direta ou indiretamente".

### D8 — Códigos de saída

- **0**: sucesso.
- **1**: erro de pipeline (alvo inexistente, fork falhou, JSON inválido,
  arquivo não encontrado, etc.).
- **2**: erro de args/validação (fonte ou alvo não informados; conflitos
  via clap).

### D9 — Helper de teste com contador atômico (problema de paralelismo)

Inicial, `escrever_json_temp` usou só `process::id()` como discriminador →
todos os testes paralelos escreviam no mesmo arquivo, causando corridas.
Corrigido com `AtomicUsize` global, dando nome único por chamada
(`g_0.json`, `g_1.json`, ...). **Descoberta de teste**: nunca subestimar
o paralelismo do `cargo test` default; arquivos temporários precisam de
unique-per-call, não unique-per-process.

### D10 — `lente_infra` como dev-dependency do CLI

Só os testes de `erro.rs` constroem variantes específicas de
`ErroAdaptador::JsonInvalido` para verificar a tradução. Em produção, o
CLI **não** importa `lente_infra` — desce tudo via `lente_wiring`
(coerente com o desenho L2 → L4). Dev-dep marca isso explicitamente.

## Mudança no `Raio` que NÃO precisou acontecer

O prompt esperava que o `saida.rs` mostrasse "Classificação" — e ela já
existe no `Raio` como enum `Classificacao` (laudo 0002). Bom. Esperava
também "Impacto direto" e "Transitivo" — também já existem (`uses_entrada`
e `montante.len()`). **Nenhuma mudança no `lente_core`** foi necessária.
Coerente com o desenho da cascata: o L2 só consome o que o L1 já tem.
